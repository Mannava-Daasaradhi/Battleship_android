=========================================================
ROOT CAUSE (single file, single rule)
=========================================================

FILE:  database.rules.json
NODE:  games/$gameId/meta  (.write rule)

The current meta .write rule allows writing ONLY when ONE of:
  (a) meta node does not yet exist (!data.exists())           → true only on game creation
  (b) caller is the HOST (data.child('hostUid') === auth.uid)
  (c) claiming guestUid slot (guestUid was null, now = my uid) → one-time write, only for that field

THE GUEST UID IS NEVER ALLOWED TO WRITE TO meta AFTER CLAIMING THEIR SLOT.

This silently (PERMISSION_DENIED caught, logged as a warning) blocks FOUR critical
operations that the GUEST device must perform at runtime:

  1. joinGame() Step 7 — advance status "waiting" → "setup"
     Called by: GUEST in MatchmakingRepository.joinGame()
     Line: metaRef.child(STATUS).setValue(STATUS_SETUP).await()
     Effect: game stays in "waiting" forever — WaitingForOpponent screen
             never advances to ship placement on both devices.
             (Currently non-fatal catch swallows the error silently)

  2. submitShipPlacement() bothReady check — advance status "setup" → "battle"
     Called by: whoever submits ships LAST. If GUEST submits last → DENIED.
     Effect: game is permanently stuck on the "Waiting for opponent" shimmer screen.
             Users call this a "crash". This is the primary reported bug.

  3. flipTurn() — write meta/currentTurn after resolving an opponent shot
     Called by: the DEFENDER after resolving each incoming shot.
             The GUEST is the defender on HOST's turn.
     Effect: turn never flips back to the GUEST. HOST fires, GUEST can never fire.
             Game deadlocks after the very first shot.

  4. claimVictory() — write meta/winner + meta/status="finished"
     Called by: whoever wins (could be GUEST)
     Effect: GUEST cannot ever formally end the game. HOST device never navigates
             to GameOverScreen if GUEST is the winner.

=========================================================
THE FIX — ONE FILE ONLY
=========================================================

FILE: database.rules.json

Add ONE condition to the meta .write rule:
  data.child('guestUid').val() === auth.uid

This allows the guest — once their UID has been written into meta/guestUid —
to write any field within meta. This is safe because:
  - Condition only triggers when the caller's UID EQUALS the stored guestUid
  - An attacker cannot write their UID as guestUid first (that is condition 4,
    only valid when guestUid is null)
  - Once a legitimate participant, they can update status/currentTurn/winner
  - Only authenticated Firebase users can trigger this path

BEFORE (broken):
  ".write": "auth != null && (
    !data.exists() ||
    data.child('hostUid').val() === auth.uid ||
    (
      data.child('guestUid').val() === null &&
      newData.child('guestUid').val() === auth.uid
    )
  )"

AFTER (fixed):
  ".write": "auth != null && (
    !data.exists() ||
    data.child('hostUid').val() === auth.uid ||
    data.child('guestUid').val() === auth.uid ||
    (
      data.child('guestUid').val() === null &&
      newData.child('guestUid').val() === auth.uid
    )
  )"

=========================================================
SCENARIO WALKTHROUGH POST-FIX
=========================================================

Scenario A — HOST submits ships last:
  GUEST submits → bothReady=false → no status change (no write attempted)
  HOST submits → bothReady=true → HOST writes status="battle" → SUCCEEDS (host always could)
  ✅ Battle starts

Scenario B — GUEST submits ships last:
  HOST submits → bothReady=false → no status change
  GUEST submits → bothReady=true → GUEST writes status="battle" → SUCCEEDS (rules now allow)
  ✅ Battle starts

Scenario C — Both submit simultaneously (race):
  Both see bothReady=true. Both read currentStatus="setup". Both attempt write="battle".
  Firebase handles concurrent writes; "battle" is idempotent — writing it twice is harmless.
  ✅ Battle starts

Scenario D — GUEST wins battle:
  GUEST calls claimVictory() → writes meta/winner=guestUid, meta/status="finished"
  SUCCEEDS (rules now allow) → both devices navigate to GameOverScreen ✅

Scenario E — flipTurn during battle (GUEST is defender):
  HOST fires. GUEST's device resolves the shot. GUEST calls flipTurn(nextTurn=GUEST).
  SUCCEEDS (rules now allow) → meta/currentTurn = guestUid → GUEST's turn ✅
